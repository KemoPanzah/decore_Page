    Distribución de software de Autodesk con el instalador ODIS y PowerShell evitando el problema del doble salto

    Desde que Autodesk cambió su sistema de despliegue al instalador ODIS, todos los despliegues desatendidos a través de la cuenta SYSTEM siguen ejecutándose sin fin y sin código de éxito.

    El mensaje de error exacto en el registro ODIS dice:

    2023-06-15T15:17:04.012 [DDA: 2196, single] [Installer INFO] [ Autodesk::DDA::SdkAgent::Listen::<lambda_8f7c2cafda285d6f3324fbe86a24bae4>::operator () ] [IPC] ChannelWin::Listen: WaitForSingleObject timeout. error_code: 997, pipe_name: \\.\pipe\adsk_dda_sdk

    Al cabo de un rato, el servicio de instalación también parece terminar, pero ya no envía un código de salida.

    He probado muchas suites de despliegue, pero el resultado es siempre el mismo cuando se utiliza la cuenta SYSTEM.

    Para resolver el problema temporalmente hasta que Autodesk reconozca y solucione el problema, ahora he desarrollado soluciones que me ayudan aquí en el trabajo.

    Mi planteamiento previsto era realizar la distribución a través de PowerShell, pero me topé con otro obstáculo que conlleva la autenticación Kerberos: el llamado problema del doble salto, ya que mis paquetes de software se almacenan en recursos compartidos CIFS.

    Kerberos no permite que se lleve a cabo una autenticación posterior a través de este punto final a otros destinos remotos después de la autenticación en un PC remoto. Así que no hay segundo salto. Para evitar el problema, hay pocas pero complicadas maneras o una que todo el mundo dirá ahora, "Oh Dios mío, no hagas eso", pero voy a lanzarla ahí fuera.

    La solución es utilizar CredSSP sólo temporalmente como una conexión punto a punto y desactivarlo de nuevo después del despliegue. Bajo ninguna circunstancia debe permitirse en toda la empresa.

    Info

    CredSSP no tiene restricciones de salto único. Las autenticaciones pueden seguir utilizándose en una sesión CredSSP. Si esto se permite en toda la empresa y de forma permanente, puede dar lugar a problemas de seguridad considerables.

    Aquí está el script para todos aquellos que se enfrentan al mismo problema.

    La secuencia de comandos que aquí se describe es deliberadamente plana para facilitar su comprensión y para que usted pueda ampliarla de la forma más sencilla posible.

    Info

    Tenga en cuenta que CredSSP sólo se utiliza temporalmente en este ejemplo y se desactiva de nuevo después de la implementación. Además, sólo se delega el equipo remoto especificado y no toda la red.

    Tras una validación exhaustiva, este enfoque ha demostrado ser una forma segura y sencilla de lograr el objetivo deseado.

    Warning

    Este script debe ejecutarse con derechos administrativos para activar y desactivar la autenticación CredSSP.

    # Define the parameters with your own values

param (
[string]$Computer = "<remote-computer>",
[string]$Name = "Autodesk Revit 2024",
[string]$Installer = '<share>\Autodesk Revit 2024\image\Installer.exe',
[string]$Argument = '"<share>\Autodesk Revit 2024\image\Installer.exe" -i deploy --offline_mode -q -o "<share>\Autodesk Revit 2024\image\Collection.xml" --installer_version "2.9.0.31"'
)

# Prompt for user credentials
$cred = Get-Credential -Message "Enter your credentials"

try {
    # Enable CredSSP on the client
    $null = Enable-WSManCredSSP -Role Client -DelegateComputer $Computer -Force

    # Enable CredSSP on the server (if not already enabled)
    Invoke-Command -ComputerName $Computer -Credential $cred -ScriptBlock {
        $null = Enable-WSManCredSSP -Role Server -Force
    }

    # Create a new PSSession with CredSSP authentication
    $session = New-PSSession -ComputerName $Computer -Credential $cred -Authentication Credssp

    # Execute the script on the remote computer
    Invoke-Command -Session $session -ScriptBlock {
        param ($Name, $Installer, $Argument)

        # Output a message before starting the installation process
        Write-Host "$Name is being installed..."

        # Start the installation process
        $process = Start-Process -FilePath $Installer -ArgumentList $Argument -PassThru

        # Output the process ID
        Write-Host "The installation process has started. Process ID: $($process.Id)"

        # Wait for the installation process to complete
        $process.WaitForExit()

        Write-Host "The installation process completed with exit code $($process.ExitCode)."

    } -ArgumentList $Name, $Installer, $Argument
}
finally {
    # Close the PSSession
    if ($session) {
        Remove-PSSession -Session $session
    }

    # Disable CredSSP on the server
    Invoke-Command -ComputerName $Computer -Credential $cred -ScriptBlock {
        Disable-WSManCredSSP -Role Server
    }

    # Disable CredSSP on the client
    Disable-WSManCredSSP -Role Client
}

    El script puede ser llamado con parámetros o puede introducir directamente sus valores específicos. La información necesaria para el instalador y los argumentos se encuentran en el script de instalación correspondiente una vez creada la imagen. Un nombre típico para una instalación de Revit podría ser 'Instalar Autodesk Revit 2024.bat', por ejemplo.

    Los parámetros de instalación y desinstalación silenciosa también se almacenan allí. Estos parámetros permiten instalar y desinstalar el software sin supervisión, lo que resulta especialmente ventajoso en entornos informáticos de gran tamaño.

    Para iniciar una sesión PowerShell utilizando CredSSP, necesitamos las credenciales de un usuario con los derechos necesarios para realizar instalaciones. En la mayoría de los casos, se trata del administrador del dominio. Aparece un cuadro de diálogo en el que puede introducir la información de inicio de sesión.

    try {
    # Enable CredSSP on the client
    $null = Enable-WSManCredSSP -Role Client -DelegateComputer $Computer -Force

    # Enable CredSSP on the server (if not already enabled)
    Invoke-Command -ComputerName $Computer -Credential $cred -ScriptBlock {
        $null = Enable-WSManCredSSP -Role Server -Force
    }

    Aquí CredSSP se activa tanto en el cliente como en el servidor. Es útil utilizar los términos "distribuidor" (cliente) y "destinatario" (servidor) para definir los roles con mayor claridad. El bloque try incluye todas las acciones potencialmente propensas a errores y puede personalizarse según sea necesario. Para empezar, esta estructura es sencilla y robusta.

    Aquí, el script se ejecuta en el ordenador remoto. El nombre, el instalador y los argumentos se pasan como parámetros. El bloque try termina en este punto.

    finally {
    # Close the PSSession
    if ($session) {
        Remove-PSSession -Session $session
    }

    # Disable CredSSP on the server
    Invoke-Command -ComputerName $Computer -Credential $cred -ScriptBlock {

    El bloque Finally cierra la sesión PowerShell y desactiva CredSSP en el servidor y en el cliente. El bloque Finally siempre se ejecuta, independientemente de si el bloque Try tuvo éxito o no. Tenga en cuenta que los errores también se pueden interceptar mediante un posible bloque de excepción. Dejo esto al lector ingenioso.

    Este script sirve ahora de base para mis propias ampliaciones. Ahora trabajaré en el desarrollo de un script superordinado para controlar y supervisar las instalaciones en grandes cantidades.
